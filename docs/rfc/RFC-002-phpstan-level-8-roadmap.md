# RFC-002: Дорожная карта повышения PHPStan-уровня до Level 8

| Поле            | Значение                                      |
|-----------------|-----------------------------------------------|
| Статус          | Proposed                                      |
| Дата            | 2026-05-05                                    |
| Автор           | Ruslan Kabirov                                |
| Затронутые файлы | `phpstan.neon`, `phpstan-baseline.neon`, все файлы `src/` |

## Мотивация

Текущий уровень статического анализа PHPStan в проекте — **level 5**. Это минимальный уровень, который ловит очевидные ошибки (несуществующие классы, вызовы несуществующих методов, неверные типы аргументов). Однако он пропускает целый класс ошибок, критичных для библиотеки, используемой в production:

### Что ловит Level 5
- Несуществующие классы и методы
- Неверное количество аргументов
- Очевидные несоответствия типов (`string` передан вместо `int`)

### Что НЕ ловит Level 5 (но ловит Level 8)
- **Nullable-ошибки**: вызов методов на `null` без проверки
- **Неверные return types**: метод обещает `string`, возвращает `?string`
- **Mixed-параметры**: передача `mixed` в типизированный аргумент
- **Дженерики**: `@template` аннотации, обобщённые коллекции, `@param iterable<string, Foo>`
- **Несовместимые array-формы**: `array{foo: string}` vs `array<string, mixed>`
- **Dead code**: недостижимые ветки, неиспользуемые переменные
- **Некорректные type-narrowing**: `instanceof` не сужает тип в некоторых сценариях

Для библиотеки с `EntityFactory`, `EntityPersister` и `UnitOfWork`, где типы данных критичны (сущность vs. прокси vs. value object vs. скаляр), уровень 5 — это недостаточная защита.

## Текущее состояние

```
# phpstan.neon
parameters:
    level: 5
    paths:
        - src
        - tests
```

Наличие `phpstan-baseline.neon` указывает на то, что некоторые ошибки уже подавлены. Это значит, что реальные проблемы уже есть, но скрыты.

## Предлагаемое решение

Поэтапное повышение уровня с 5 до 8, по одному уровню за release-цикл. Каждый этап включает:

1. Повышение уровня на 1.
2. Исправление всех найденных ошибок (или добавление в baseline с TODO-комментарием).
3. Обновление CI-матрицы.
4. Коммит с пометкой в changelog.

### Фаза 1: Level 5 → Level 6

**Фокус**: Boolean types,严格的 null-checking

Level 6 начинает требовать явную обработку nullable-значений. Типичные ошибки, которые появятся:

```php
// Level 5 пропускает:
$metadata = $this->getClassMetadata($entityName);
$adapterClass = $metadata->getEntityDataAdapterClass();
// Если getEntityDataAdapterClass() возвращает ?string — Level 6 потребует проверку

// Level 6 потребует:
if ($adapterClass === null) {
    throw new RuntimeException('No adapter class defined');
}
$adapter = new $adapterClass();
```

**Ожидаемое количество новых ошибок**: 30-50.

**Оценка времени**: 1-2 дня.

**Типичные исправления**:
- Добавить `if ($x === null)` проверки перед доступом к nullable-свойствам.
- Уточнить return types методов, которые всегда возвращают значение, но типизированы как `?T`.
- Заменить `@return mixed` на конкретные типы в docblock-ах.

### Фаза 2: Level 6 → Level 7

**Фокус**: Объединение типов, array-shape анализ

Level 7 начинает анализировать массивы как структуры (`array{key: type}`) и union types. Это особенно актуально для `$metadata` array (см. ADR-002), но даже без его замены — многие методы возвращают array с известной структурой.

```php
// Level 7 начнёт проверять:
private function getFieldOptions(string $fieldName): array
{
    return $this->metadata['fields'][$fieldName] ?? [];
    // Type of ?? [] — array. Но что внутри? Level 7 потребует уточнения.
}
```

**Ожидаемое количество новых ошибок**: 40-80.

**Оценка времени**: 2-3 дня.

**Типичные исправления**:
- Добавить `@return array<string, mixed>` или `@return array{type: FieldTypeEnum, column: string, ...}` docblock-и.
- Уточнить типы параметров Closure в `EntityProxyTrait`.
- Добавить `@param` и `@return` для callback-параметров в `PersistentCollection`.

### Фаза 3: Level 7 → Level 8

**Фокус**: Полная типовая Strictness, дженерики

Level 8 — максимальная strictness. Все `mixed` должны быть обоснованы, все дженерики аннотированы.

```php
// Level 8 потребует:
/**
 * @param class-string<T> $className
 * @return T
 * @template T of object
 */
public function find(string $className, mixed $id): ?object

// Вместо текущего:
public function find(string $className, mixed $id): ?object
```

**Ожидаемое количество новых ошибок**: 50-100.

**Оценка времени**: 3-5 дней.

**Типичные исправления**:
- Добавить `@template` аннотации в `EntityRepository`, `EntityPersister`, `EntityFactory`.
- Аннотировать `PersistentCollection` как `@extends ArrayObject<int, object>`.
- Уточнить `class-string` параметры вместо `string`.
- Добавить `@param callable(mixed): mixed` для callback-ов.
- Устранить все оставшиеся `mixed` там, где возможен конкретный тип.

## Работа с Baseline

На каждом этапе неизбежно появятся ошибки, которые невозможно или нецелесообразно исправить сразу (например, в устаревшем API, который планируется удалить). Для таких случаев:

1. Ошибка добавляется в `phpstan-baseline.neon` с комментарием `// TODO: fix after ADR-002 migration` или аналогичным.
2. Каждая запись в baseline должна иметь срок — номер release, к которому она должна быть устранена.
3. Baseline не должен расти — на каждом этапе количество записей в baseline должно уменьшаться или оставаться стабильным.

Формат baseline с TODO:

```neon
# phpstan-baseline.neon
parameters:
    ignoreErrors:
        -
            message: '#Method .+ returns mixed#'
            path: src/Persistence/EntityPersister.php
            comment: 'TODO: fix after ADR-002 (typed metadata DTOs)'
            deadline: '2.0.0'
```

> **Примечание**: PHPStan не поддерживает `comment` и `deadline` нативно, но их можно добавить как документацию в неоне или в отдельном файле `baseline-todo.md`.

## Влияние на производительность CI

Повышение уровня PHPStan увеличивает время анализа:

| Level | Время анализа (ориентировочно) |
|-------|-------------------------------|
| 5     | ~15 секунд                    |
| 6     | ~20 секунд                    |
| 7     | ~30 секунд                    |
| 8     | ~45 секунд                    |

Это приемлемо для CI. Если время станет проблемой, можно разделить анализ на параллельные джобы по namespace.

## Критерии успеха

- [ ] PHPStan level 8 проходит без ошибок
- [ ] Baseline содержит менее 10 записей
- [ ] Все записи в baseline имеют TODO с release-дедлайном
- [ ] CI-матрица включает PHPStan на каждом PR
- [ ] `phpstan.neon` содержит `level: 8` и `treatPhpDocTypesAsCertain: true`

## Альтернативы

### Альтернатива 1: Сразу Level 8

Поднять уровень до 8 одним коммитом, загнав все ошибки в baseline, и постепенно исправлять.

**Плюс**: сразу видна полная картина проблем.
**Минус**: baseline на 200+ записей демотивирует. Нет инкрементального прогресса. Сложно ревьюить PR с сотнями изменений.

### Альтернатива 2: Psalm вместо PHPStan

Перейти на Psalm, который традиционно строже в работе с массивами и дженериками.

**Плюс**: лучший анализ array shapes и дженериков.
**Минус**: два инструмента вместо одного — это двойная нагрузка на CI и на разработчиков. Psalm и PHPStan иногда расходятся в интерпретации типов. Для библиотеки с экосистемой (Symfony bundle) лучше оставаться на PHPStan, который является стандартом в Symfony-мире.

### Альтернатива 3: Оставить Level 5

Не тратить время на повышение уровня.

**Минус**: библиотека, претендующая на production-использование и совместимость с Doctrine, должна соответствовать стандартам качества Doctrine. Doctrine ORM использует PHPStan level 8. Отставание в качестве статического анализа снижает доверие к библиотеке.
